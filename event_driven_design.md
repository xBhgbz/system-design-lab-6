# Event-Driven Architecture Design

## Обзор системы

Система управления бронированием отелей используется для бронирования отелей. Сам процесс бронирования должен происходить синхронно, с блокировками, чтобы если 2 клиента одновременно начали бронировать последний номер в отеле, то один получил успех, а второй ошибку. Об ошибке бронирования клиента кадо сразу же оповестить, поэтому с бронированиями Event-Driven архитектуру сделать не получится, так как важна консистентность данных в реальном времени.

При этом добавление и отображение этого отеля может происходить с небольшой задержкой, что позволяет использовать Eventual Consistency, но это не такое частое действие, чтобы ради него использовать Event-Driven архитектуру.

Итого получается, что ивенты можно использовать для нотификации отелей о новых бронированиях, но сами бронирования должны работать синхронно. В процессе броинрования или его отмены должен отправляться ивент, получателем которого могут быть отели.

## Компоненты архитектуры

События:
- Создание бронирования.
- Отмена бронирования.

Они инициируются соответствующими командами.

## События в системе

Продюссером событий будет сервис API, а потребителем тестовый сервис consumer, который будет печатать сообщения в консоль.

### Event: `booking.created`

При POST запросе на `/api/bookings`
```json
{
  "event_type": "booking.created",
  "event_id": "uuid",
  "timestamp": "2026-05-19T12:34:56Z",
  "data": {
    "booking_id": 1,
    "user_id": 2,
    "hotel_id": 5,
    "start_date": "2026-06-01",
    "end_date": "2026-06-05"
  }
}
```

### Event: `booking.cancelled`

При DELETE запросе на `/api/bookings/{id}`
```json
{
  "event_type": "booking.cancelled",
  "event_id": "uuid",
  "timestamp": "2026-05-19T12:34:56Z",
  "data": {
    "booking_id": 1,
    "user_id": 2,
    "hotel_id": 5
  }
}
```

В качестве брокера сообщений я выберу RabbitMQ из-за простоты настройки и эксплуатации для небольших проектов. 
Гарантии доставки для всех сообщений должны быть `At least once`, чтобы клиенты точно получили информацию о бронировании. Для дедубликации на стороне клиента я добавлю в сообщения ключ идемпотентности.

## CQRS (Command Query Responsibility Segregation)

В моем случае применить невозможно. Основные операции должны обрабатываться синхронно.
